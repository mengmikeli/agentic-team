## Security Analysis

### Threat Model

This is a **local CLI tool** (`agt report <feature>`) that:
- Reads a local STATE.json file
- Renders a markdown report to stdout or a local file
- Is invoked by the user who owns the machine

Primary threat vectors: **path traversal** (reading/writing outside intended directory) and **input injection** (malformed STATE.json data crashing or corrupting output).

### Input Validation at Trust Boundary (`cmdReport`)

| Check | Location | Mechanism | Verified |
|-------|----------|-----------|----------|
| Missing feature name | report.mjs:151 | `!featureName` → exit 1 | Test at line 470-475 |
| Path traversal | report.mjs:163 | `basename(featureName) !== featureName` rejects `/`, `..`, etc. | Tests at lines 606-625 |
| Dot entries | report.mjs:163 | Explicit `.` and `..` rejection | Tests at lines 613-625 |
| Invalid `--output` value | report.mjs:157 | Only `"md"` accepted | Tests at lines 588-601 |
| Missing feature directory | report.mjs:171 | `existsSync` check → exit 1 | Test at line 479-484 |
| Missing STATE.json | report.mjs:178 | `readState` returns null → exit 1 | Test at line 488-493 |

### Path Traversal — Deep Verification

`basename()` behavior confirmed for attack payloads:
- `../../etc` → basename = `etc` ≠ `../../etc` → **rejected**
- `/etc/passwd` → basename = `passwd` ≠ `/etc/passwd` → **rejected**
- `foo/bar` → basename = `bar` ≠ `foo/bar` → **rejected**
- `normal-name` → basename = `normal-name` == `normal-name` → **passes** (correct)
- `.` → explicitly rejected
- `..` → explicitly rejected

File write path is constrained: `join(_cwd(), ".team", "features", featureName, "REPORT.md")` — no user-controlled path components beyond the validated feature name.

### No Shell Injection Surface

The feature name is **never** passed to a shell command. All usage is through Node.js `path.join()`, `existsSync()`, `readFileSync()`, and `writeFileSync()`. No `exec`, `spawn`, or template strings with shell evaluation.

### JSON.parse Safety

`readState` (util.mjs:190-198) uses `JSON.parse()` which:
- Creates plain objects (no prototype pollution via `__proto__`)
- Cannot execute code
- Returns null on parse failure (caught by the null check at report.mjs:178)

### Data Rendering (buildReport)

`buildReport` renders STATE.json data into markdown. Security-relevant observations:

1. **`escapeCell`** (report.mjs:8-10): Strips `\r\n` and escapes `|` — prevents markdown table breakage from task titles
2. **`Number.isFinite`** (report.mjs:26): Guards against NaN leaking into duration display from malformed ISO dates
3. **`toFixed(4)`** (report.mjs:73,77): Called only after null check on `costUsd` — but does not guard against non-numeric types

### No Secrets Exposure

The report renders: feature name, task IDs/titles/statuses, attempt counts, gate verdicts, cost numbers, and timestamps. No credentials, tokens, API keys, or PII are present in the data model or output.

## Security Findings

🔵 `bin/lib/report.mjs:158` — Error message reflects `outputVal` (user CLI arg) to stderr without sanitizing ANSI escape sequences. No real threat: user controls their own terminal, and this is a local CLI tool.

🔵 `bin/lib/report.mjs:73` — `costUsd` is null-checked but not type-checked. A corrupted STATE.json with `"costUsd": "string"` would throw `TypeError` on `.toFixed(4)`. Robustness issue only — not exploitable since STATE.json is machine-generated by the harness.

🔵 `bin/lib/report.mjs:8` — `escapeCell` handles `|` and `\r\n` but not other markdown syntax characters (`[`, `` ` ``). Cosmetic rendering issue in REPORT.md, no security impact since file is locally generated.

## Verdict Rationale

The implementation has **solid security hygiene** for a local CLI tool:

1. The input validation boundary is correctly placed at `cmdReport` entry — all user-facing input is validated before use
2. Path traversal is the primary realistic threat and is well-guarded with `basename()` check plus explicit `.`/`..` rejection, backed by tests
3. No user input reaches shell execution or `eval`
4. File operations are scoped to the expected `.team/features/` directory
5. JSON parsing is safe by design (`JSON.parse` cannot execute code)
6. All three findings are suggestions (🔵) with no realistic attack surface

**PASS** — No critical or warning security issues. Three optional suggestions filed, none blocking merge.
